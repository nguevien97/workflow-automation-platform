Complex Domain Modeling

A Senior/Staff Engineer Interview Session — Object-Oriented Design & Domain-Driven Design

Enterprise Workflow Automation Platform

The Scenario

Think Zapier or Microsoft Power Automate, but for enterprise. Design the core domain for a platform where non-technical business users build their own automated workflows.

Here are the business requirements:

  1. A Workspace belongs to an organization. Users within the workspace create Workflows. A workflow is a chain of Steps that execute in sequence when triggered.
  2. Each step is one of three types:
    • Trigger: the event that starts the workflow (e.g., "when a new row is added to Google Sheets," "when an email arrives in Gmail," "when a form is submitted"). Can be event-based or schedule-based ("every Monday at 9am").
    • Action: do something (e.g., "send a Slack message," "create a Jira ticket," "insert a row into a database").
    • Condition: a branching point (e.g., "if the email subject contains 'urgent', go to Step A, otherwise go to Step B"). Conditions can create parallel branches that later merge.
  3. Steps connect to external services through Integrations. An integration holds the OAuth credentials and configuration for a specific service (e.g., the workspace's Slack integration). Integrations are per-workspace — shared by all users and all workflows in that organization. Multiple workflows can share the same integration.
  4. The data mapping problem: each step produces an output (e.g., the Trigger "new email" produces {sender, subject, body}). Subsequent steps can reference outputs from any previous step. A user might configure an Action step like: "send a Slack message where the text is New urgent email from {{trigger.sender}} about {{trigger.subject}}." These template references must be validated — you can't reference an output field that doesn't exist, and you can't reference a step that isn't upstream in the workflow graph.
  5. Versioning: when a user modifies a workflow, currently running executions must complete using the old version. New triggers use the new version. Multiple versions can be executing simultaneously.
  6. Execution: when a workflow is triggered, the system creates an Execution that walks through the steps. Each step execution can succeed, fail, or timeout. If a step fails, the user can configure one of three strategies: stop the workflow, skip and continue, or retry N times then stop.
  7. Rate limiting: external services have API rate limits. If 50 workflows all try to send a Slack message at the same second, the platform must queue and throttle them. Rate limits are per-integration, not per-workflow.
  8. Webhook Triggers with Deduplication: External services can trigger workflows via webhook. But webhooks are unreliable — the sender might retry, delivering the same event 3 times. The system must deduplicate webhook deliveries within a 5-minute window using an idempotency key. Additionally, webhook payloads must be validated against a user-defined JSON schema before triggering the workflow. Invalid payloads are logged but do not create executions.
  9. Execution Timeout and Cost Budgets: Each workflow has a maximum execution timeout (default 30 minutes, configurable up to 24 hours for long-running data pipelines). Each workspace has a monthly execution budget — measured in total step executions. A free workspace gets 1,000 step executions per month. A paid workspace gets 50,000. When the budget is exhausted, scheduled triggers stop firing, but webhook triggers still create executions that immediately fail with a "budget exceeded" error (so the webhook sender gets feedback). Budget tracking must be accurate under concurrent execution — 20 workflows finishing simultaneously must not overshoot the budget.
  10. Secrets and Variable Scoping: Workspaces have environment variables (e.g., API_KEY, DATABASE_URL) that can be referenced in step configurations using {{env.API_KEY}}. Variables have scopes: workspace-level (shared by all workflows), workflow-level (override workspace defaults), and execution-level (passed in at trigger time, e.g., webhook payload fields). Narrower scope wins. Secrets (variables marked as sensitive) must never appear in execution logs — the system must redact them from step outputs before persisting.
  11. Integration Definition Framework: Integrations are not hardcoded. The platform supports a plugin architecture where each integration is defined declaratively. An integration definition contains:
    • Authentication spec: the type of auth (OAuth2, API key, Basic Auth, custom token refresh), the endpoints for auth flow, token refresh logic, and what headers/query params the token gets attached to. OAuth2 alone has variations: authorization code flow, client credentials flow, PKCE. Some services (like Shopify) require per-store OAuth where each customer has a different OAuth app. Others (like Slack) use workspace-level tokens.
    • Action catalog: each integration exposes a set of actions. Slack might expose: send message, create channel, list users, upload file. Each action has an input schema (what fields the user must configure), an output schema (what fields the action produces for downstream steps), and an HTTP spec (method, URL template, headers, body template, pagination strategy). For example, "send Slack message" has input {channel: string, text: string}, output {messageId: string, timestamp: string}, and HTTP spec POST https://slack.com/api/chat.postMessage with the token in the Authorization header and the body as JSON.
    • Trigger catalog: each integration can expose triggers. Two types: webhook-based (Slack sends an event to your endpoint when a message is posted) and polling-based (every 5 minutes, call the Google Sheets API and check if new rows were added since last poll). Polling triggers need to track a cursor — a timestamp, page token, or row ID that marks "where we left off." The cursor must persist across polls and be per-workflow, not per-integration.
    • Response mapping: external APIs return wildly different response structures. Slack wraps everything in {ok: true, data: {...}}. Google returns {items: [...], nextPageToken: "..."}. Some APIs return 200 with an error in the body. The integration definition includes a response transformer — a declarative mapping that extracts the actual data, detects errors, and normalizes the output to match the declared output schema.
    • Pagination: some actions return paginated results. "List all Jira issues" might return 50 at a time with a cursor. The integration definition specifies the pagination strategy: cursor-based, offset-based, or page-number-based. The execution engine must auto-paginate and aggregate all pages into a single output array before passing it to the next step. This ties directly into the For-Each requirement — "list all Jira issues" producing 3,000 results flows into a For-Each loop.
  12. Custom Integrations by Users: Users can create their own integrations within their workspace. They define the auth method, base URL, and individual actions with HTTP specs and schemas. This is essentially a visual API client builder. A user might create a "My Company Internal API" integration with actions like "create employee record" and "fetch department list." Custom integrations go through the exact same execution path as built-in integrations. The execution engine doesn't know or care whether it's hitting Slack's API or a user's internal service. This means your integration abstraction must be clean enough that the engine only speaks to a uniform interface.
  13. Webhook Management: For webhook-based triggers, the platform must manage webhook registration with external services. When a user creates a workflow with a Slack "new message" trigger, the platform must call Slack's API to register a webhook URL. When the workflow is deleted or the trigger is changed, the platform must deregister the webhook. If the registration fails (permissions issue, Slack is down), the workflow cannot be activated. Some services require webhook signature verification — Slack signs payloads with HMAC-SHA256, Stripe uses a different signing scheme, GitHub uses yet another. The integration definition includes the signature verification method and secret location. Webhook URLs must be unique per workflow. If two workflows both trigger on "new Slack message in #general," they each get separate webhook registrations and separate incoming URLs. The platform routes incoming webhooks to the correct workflow based on the URL path.
  14. OAuth Token Lifecycle: OAuth tokens expire. Refresh tokens sometimes expire too. Some providers (Google) give you a refresh token once and never again — if you lose it or it expires, the user must re-authorize. The platform must proactively refresh tokens before they expire. If a refresh fails, all workflows using that integration must pause (you designed this already in the original scenario), and the user must be notified to re-authorize. Token refresh must be serialized per integration instance. If 10 workflows simultaneously discover the token is expired, they must not all attempt to refresh concurrently — that would send 10 refresh requests to the provider, 9 of which will fail (or worse, the provider rotates the refresh token on each use and now you have a race condition where the first refresh invalidates the token the second refresh is using).
  15. Rate Limit Discovery and Adaptation: Different APIs express rate limits differently. Slack returns X-RateLimit-Remaining and X-RateLimit-Reset headers. GitHub returns X-RateLimit-Limit, X-RateLimit-Remaining, X-RateLimit-Reset. Some APIs (like Notion) return 429 with a Retry-After header but no proactive headers. Some APIs have multiple rate limit tiers (Shopify: 40 requests per app per store per minute, plus a daily limit of 10,000). The integration definition declares the rate limit strategy: header-based (read remaining/reset headers from every response and adjust), reactive (just handle 429s with exponential backoff), or configured (hardcoded limit set by the user). The rate limiter must adapt in real-time — if the response says "3 requests remaining, resets in 45 seconds," queue the next requests accordingly.

The workflow is always a Directed Acyclic Graph (DAG) — no cycles allowed. A workflow has exactly one Trigger step as its entry point. Multiple executions of the same workflow can overlap (e.g., a scheduled workflow running every 15 minutes might still be processing the previous run when the next one triggers).

Why this is brutal:

The workflow is a user-defined directed graph with branching and merging. The data mapping creates a dependency chain where each step's configuration depends on the schema of upstream steps — change a trigger's output schema and you might break 15 downstream steps across 3 branches. Versioning means the "current" workflow and the "executing" workflow can be different objects. Rate limiting spans across all workflows that share an integration.

Architecture & Performance Pressure:

  1. Pressure 1 — Execution Engine Hot Path: A large workspace has 200 active workflows, 50 of which are scheduled to run every minute. At the top of each minute, 50 workflows trigger simultaneously. Each workflow averages 8 steps. That's 400 step executions per minute sustained, with spikes of 50 concurrent workflow starts. The execution engine must process these without falling behind — if minute-2's triggers fire while minute-1's executions are still running, they must not queue indefinitely. Design the step execution pipeline so it scales horizontally. Think about whether WorkflowExecution and ActionExecution can be processed by different workers, and how the event flow between them handles backpressure.
  2. Pressure 2 — Rate Limiter Under Contention: 30 workflows in the same workspace all use the Slack integration. Slack's API allows 50 requests per minute per workspace token. During a spike, 30 workflows each try to send a Slack message within the same 10-second window. The rate limiter must queue excess requests and drain them at the allowed rate without dropping any, while providing backpressure to the execution engine so step executions wait rather than timeout. Design the IntegrationRequest aggregate and its queue so it handles this gracefully. Consider: is rate limiting per-integration-instance (one Slack token) or per-integration-type? What happens when queued requests exceed the step's timeout?
  3. Pressure 3 — Execution Log Query Performance: Users view execution history — filtered by workflow, status, date range, and duration. For a workspace with 100,000 executions per month, the UI must return paginated results within 500ms. Each execution has 5-15 step execution records with inputs, outputs (redacted), duration, and status. Design your read model and storage strategy. Think about whether execution logs belong in the same database as the write-side aggregates.
  4. Pressure 4 — Workflow Version Migration: A user modifies a workflow that currently has 12 active executions using the old version. New triggers must use the new version. But the user also wants to see "which version is each execution running?" in the dashboard. If a workflow has been edited 47 times, do you keep all 47 versions? How do you garbage-collect versions that have no running executions and are not the current version?
  5. Pressure 5 — For-Each with 50,000 Items: A data pipeline workflow fetches 50,000 rows from a database, then runs a For-Each that calls an external API for each row (max concurrency: 10, external API rate limit: 100 requests/minute). This single workflow execution will run for approximately 8 hours. Design the execution so it's resumable — if the worker crashes at iteration 23,000, it can pick up from where it left off without re-processing completed iterations. Think about checkpointing.
  6. Pressure 6 — Integration Hot-Swap: A built-in integration definition is updated (Slack changes their API endpoint, or adds a new action). The update must not break running executions. Running executions continue with the old definition. New executions use the new definition. This is the same versioning problem as workflows, but now applied to integrations. How do you version integration definitions, and does ActionExecution carry a snapshot of the integration definition it's using?
  7. Pressure 7 — Multi-Tenant Token Isolation: The platform hosts 10,000 workspaces. Each workspace has its own Slack token, Google token, etc. Tokens must be encrypted at rest. Token refresh must not leak one workspace's token to another. A bug in the refresh logic for Workspace A must not corrupt Workspace B's token. Design the token storage and refresh pipeline with tenant isolation as a hard constraint.
  8. Pressure 8 — Observability per Integration: The operations team needs to see: per integration type, across all workspaces, what's the current error rate, average latency, rate limit headroom, and token health. This is a cross-tenant read model. A Slack outage should be detectable within 2 minutes by seeing error rates spike across all workspaces using Slack. Design this as a separate read model that aggregates integration health without exposing tenant-specific data.

Task:

  1. Bounded contexts and aggregates — how do you slice this domain?
  2. The workflow graph with data mapping validation — how do you model the steps, branches, and template references? Where does validation of {{trigger.sender}} live?
  3. Versioning — how do you model the relationship between a workflow and its versions so that running executions are isolated from edits?
  4. Rate limiting — this spans all workflows sharing an integration. Where does this concern live?

Design

Candidate

WorkflowDefinition (supporting domain):

A WorkflowDefinition contains a tree structure of steps. Example data structure:

[
  trigger,
  Action A,
  {
    type: condition,
    condition: x,
    branches: {
      "value1": [Action B, Action C],
      "value2": [[Action F], [Action D, Action E]]
    }
  }
]

ActionDefinition — id, type: Action, integrationId, command (a symbolic field that represents what the action does, e.g. "send a Slack message," "create a Jira ticket"), failStrategy.

Design-time validation of data mappings ({{trigger.sender}}) lives inside this supporting domain.

Execution Context:

WorkflowExecution (aggregate root):

  • workflowVersionId
  • workflowDefinition (value object from supporting domain) — with getNextStep(currentStepId)
  • status: InProgress / Stop
  • stepExecutions[]

StepExecution (entity inside WorkflowExecution):

  • stepExecutionId, stepId, output, status: started/skip/stop
  • getInput() — resolves template references from upstream step outputs
  • start() → if this step is an Action, publish a domain event ActionStarted(workflowExecutionId, stepExecutionId, input)
  • For trigger and condition steps, just start the next appropriate step execution
  • Subscribes to ActionDone/ActionSkip to update status and start the next step execution

Triggers fire at the application service layer. Trigger fires = create a new WorkflowExecution then start the first step after trigger.

ActionExecution (aggregate root):

  • workflowExecutionId, stepExecutionId, retryTimes, input, output
  • execute() → if using an integration service, publish a domain event IntegrationRequested(actionExecutionId, input, integrationType, command)
  • When done → publish a domain event ActionDone/ActionSkip(workflowExecutionId, stepExecutionId, output)
  • ActionDefinition — integrations, failStrategy, command
  • Subscribes to IntegrationRequestDone/Failed

IntegrationRequest (aggregate root):

  • Subscribes to IntegrationRequested
  • Implements rate limiting here
  • Publishes IntegrationRequestDone/Failed

Key Design Decisions Explored

Versioning

Solved elegantly: By embedding WorkflowDefinition as a value object inside WorkflowExecution, each execution carries its own frozen copy of the definition. Users can edit the workflow freely — running executions are completely isolated because they hold their own snapshot.

Branch Merging

Interviewer

In your WorkflowExecution, how does getNextStep(currentStepId) work when two branches merge back into a single step? When Action C completes on Branch 1, does the merge step start immediately? Or does it wait for Branch 2 to also complete?

Candidate

When the next step after a done step is a merge step, but the other branches are not done yet, just return null and do nothing. When other branches are done, the next step will be resolved.

Since WorkflowExecution owns all StepExecution entities internally, this "are all branches done?" check is purely internal — no cross-aggregate coordination needed.

StepExecution vs ActionExecution Separation

Interviewer

You separated ActionExecution from StepExecution. That means a single step execution produces two aggregates coordinated by events. What drove that decision? Could ActionExecution's logic live inside StepExecution instead?

Candidate

StepExecution is just an entity inside WorkflowExecution for tracking step completion. WorkflowExecution is just meant to handle status. ActionExecution handles the messy reality of retries, timeouts, and integration calls.

Clean separation: WorkflowExecution is the orchestrator that tracks which steps are done and determines what's next. ActionExecution is the worker that handles retries, timeouts, and integration communication. Same principle as the Promotion Engine — the orchestrator is dumb, the workers are smart.

Integration Credential Failure

Interviewer

A workspace's Slack OAuth token expires at 2am. There are 30 workflows that use Slack, and 12 of them have scheduled triggers between 2am and 6am. All 12 executions fail with an authentication error. This isn't a transient failure that retries will fix.

Do all 12 ActionExecution aggregates burn through their retry attempts independently? Or is there a way to detect "this integration is globally broken" and pause all workflows depending on it?

Candidate

When a token expires, just pause all IntegrationRequest for that workspace. This raises a question about whether WorkflowExecution needs to update its status to Pending (or something else) accordingly. Users may cancel a Pending workflow execution. When the integration setting is updated with a new token, continue if the user didn't cancel the execution yet.

Final Architecture

WorkflowDefinition Context (supporting) — workflow graph structure, step definitions, design-time data mapping validation. Produces a WorkflowDefinition value object that gets embedded into executions.

Execution Context — WorkflowExecution aggregate root (graph traversal, step tracking, branch merging) and ActionExecution aggregate root (retry logic, failure strategies). Events: ActionStarted, ActionDone, ActionSkip.

Integration Context — IntegrationRequest aggregate root (rate limiting, external API communication, global failure detection). Events: IntegrationRequested, IntegrationRequestDone/Failed, IntegrationUnavailable, IntegrationRestored.

Execution pipeline: Trigger fires → WorkflowExecution created → StepExecution.start() → ActionStarted → ActionExecution.execute() → IntegrationRequested → IntegrationRequest handles rate limiting and API call → result cascades back up through events → next step starts.

Refinements Discussed

  • Integration Health as a Read Model: Proactively monitor token expiry timestamps. Publish IntegrationExpiring(workspaceId, integrationType, expiresIn: 30 minutes) before it actually expires, preventing wasted executions.
  • Step Output Schema as a First-Class Value Object: Formalize each step's output schema as StepOutputSchema { fields: [name, type] }. Makes validation declarative and enables autocomplete in the workflow builder UI.
  • Execution Context Split: WorkflowExecution and ActionExecution could be in separate contexts if owned by different teams with different release cycles. But if they always change together, keep them together.
