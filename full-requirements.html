1. Workspace & Workflows

A Workspace belongs to an organization. Users within the workspace create Workflows. A workflow is a Directed Acyclic Graph (DAG) of Steps that execute when triggered. A workflow has exactly one Trigger step as its entry point. Multiple executions of the same workflow can overlap (e.g., a scheduled workflow running every 15 minutes might still be processing the previous run when the next one triggers).

2. Step Types (Trigger, Action)

Each step has a StepId, StepType, Name, and NextStepId (the step that executes after this one completes; null if last in its chain). There are five step types total — Trigger and Action are covered here; Condition, Parallel, and Loop are covered in Requirement 8.

2.1 Trigger Step

The entry point of the workflow. Exactly one per workflow. Connected to an integration and a command (e.g., Slack "new message", Google Sheets "new row"). Can be event-based (webhook) or schedule-based ("every Monday at 9am").

  • IntegrationId — which integration this trigger uses
  • CommandName — which trigger command (e.g., "newMessage")
  • Configuration — trigger-specific settings (e.g., which Slack channel)
  • OutputSchema — required, declares the fields this trigger produces
  • No InputMappings (no upstream step)
  • No FailureStrategy (entry point, not a fallible step)

2.2 Action Step

Calls an external service through an integration.

  • IntegrationId — which integration this action uses
  • CommandName — which action command (e.g., "sendMessage")
  • InputMappings — dictionary of field name → literal value or template reference
  • OutputSchema — required, declares the fields this action produces
  • FailureStrategy — Stop (halt workflow), Skip (continue), or Retry(N) (retry N times then stop)

3. Integrations & Credentials

Steps connect to external services through Integrations. An integration holds the OAuth credentials and configuration for a specific service (e.g., the workspace's Slack integration). Integrations are per-workspace — shared by all users and all workflows in that organization. Multiple workflows can share the same integration.

4. Data Mapping & Template References

Each step produces an output defined by its OutputSchema. Subsequent steps can reference outputs from upstream steps using template syntax: {{stepId.fieldName}}. For example: New urgent email from {{trigger.sender}} about {{trigger.subject}}.

Template references must be validated at design time (when the workflow is saved):

  • The referenced step must exist in the workflow
  • The referenced step must be guaranteed to have completed before the current step executes (see Section 8.3 for complete rules)
  • The referenced field must exist in the referenced step's OutputSchema

5. Versioning

When a user modifies a workflow, currently running executions must complete using the old version. New triggers use the new version. Multiple versions can be executing simultaneously.

Implementation: The WorkflowDefinition is embedded as a value object snapshot inside each WorkflowExecution. Running executions are completely isolated from subsequent edits because they carry their own frozen copy of the definition.

6. Execution & Failure Strategies

When a workflow is triggered, the system creates an Execution that walks through the steps. Each step execution can succeed, fail, or timeout.

If a step fails, the user-configured failure strategy applies:

  • Stop — halt the entire workflow, mark as failed
  • Skip — mark the step as skipped (output is null), continue to the next step
  • Retry(N) — retry up to N times with backoff; if still failing after N retries, stop the workflow

7. Rate Limiting

External services have API rate limits. If 50 workflows all try to send a Slack message at the same second, the platform must queue and throttle them. Rate limits are per-integration-instance (one Slack workspace token), not per-workflow.

8. Branching, Parallel Execution & Loops

Overview

The workflow DAG supports branching (conditions), parallel execution (fork/merge), and iteration (loops). This section defines three additional step types — Condition, Parallel, and Loop — along with the DAG structure, template reference validation rules, and execution semantics.

8.1 Condition Step

A branching point that evaluates rules and routes execution to exactly one branch. Follows the Zapier "Paths" model: first match wins.

Contains an ordered list of ConditionRule value objects. Each rule has:

  • Condition — a boolean expression (e.g., "{{trigger.priority}} == 'urgent'", "{{trigger.amount}} > 1000 AND {{trigger.department}} == 'finance'")
  • BranchEntryStepId — the StepId of the first step in the branch to execute if this rule matches

Rules are evaluated top-to-bottom. The first rule whose condition evaluates to true wins, and execution goes to that rule's BranchEntryStepId.

Additional fields:

  • FallbackBranchEntryStepId — optional; if no rule matches, execution goes here. If no fallback is defined and no rule matches, the workflow fails with a "no matching condition" error.
  • NextStepId — the continuation point after the winning branch completes (the step that runs after the branch's last step finishes)

Each branch is a chain of steps linked by NextStepId. The last step in the branch has NextStepId = null. After the winning branch completes, execution continues to the condition step's own NextStepId.

Only one branch executes. There is no parallel execution from a condition step. A condition does NOT produce output. It does not have an OutputSchema, InputMappings, or FailureStrategy.

8.2 Parallel Step

A fork point that executes all branches simultaneously and waits for all to complete before continuing. This is a distinct step type from Condition.

  • BranchEntryStepIds — list of StepIds, one for the first step of each parallel branch. Must have at least two entries.
  • NextStepId — the merge point; the step that executes after all branches complete

Each branch is a chain of steps linked by NextStepId. The last step in each branch has NextStepId = null.

Failure Semantics

Each branch's steps have their own individual failure strategies (Stop, Skip, Retry). These apply at the step level within the branch.

  • If a step in a branch fails with Stop strategy: all other still-running branches are cancelled mid-execution, and the workflow fails.
  • If a step in a branch fails with Skip strategy: the step is marked as skipped with null output, the branch continues to its next step. The merge step receives null for the skipped step's output.

A parallel step does NOT have a condition expression, rules, input mappings, output schema, or failure strategy.

8.3 Loop Step

Iterates over an array from an upstream step's output and executes a sub-workflow for each item.

  • SourceArray — a template reference pointing to an array field from an upstream step (e.g., {{step1.rows}})
  • Body — a LoopBodyDefinition value object (embedded, not a separate aggregate) containing:
    • TriggerOutputSchema — the schema of a single item from the source array. Acts as the "trigger" for each iteration; body steps reference item fields via {{trigger.fieldName}}
    • Steps — the body's step chain (a mini-workflow with its own step definitions linked by NextStepId)
  • ConcurrencyMode — Sequential or Parallel(maxN)
  • IterationFailureStrategy — StopAll (cancel remaining iterations) or Skip (skip the failed iteration, continue with others)
  • NextStepId — the step after the loop completes

The loop step's output is an array of all iteration results (the output of the last step in each iteration's body).

Parent Context Access

Steps inside the loop body can reference outputs from steps upstream of the loop in the parent workflow (e.g., {{parentStep.apiKey}}). At execution time, the parent WorkflowExecution passes upstream outputs as ParentContext into each iteration's execution.

Embedded Body, Separate Execution

The loop body is embedded as a value object in the WorkflowDefinition for design-time validation. At execution time, each iteration spawns a separate WorkflowExecution with the body's definition snapshot and the parent context.

Two Levels of Failure Handling

Step-level (inside the loop body): A step inside the body fails. The step's own FailureStrategy applies first: Retry 3 times? Skip this step? Stop the body? This is resolved within the iteration's execution.

Iteration-level (on the loop step): The entire iteration's execution fails (a step with Stop strategy failed after exhausting retries). Now the loop's IterationFailureStrategy applies: StopAll remaining iterations? Skip this iteration?

Example: A loop body has Step D with FailureStrategy = Retry(3). Step D fails. It retries 3 times. Still fails. Step D's strategy after retries is Stop → the iteration fails. The loop's IterationFailureStrategy is Skip → this iteration is skipped, remaining iterations continue.

8.4 DAG Structure

All steps (including those inside condition branches and parallel branches) live in a flat dictionary within the WorkflowDefinition aggregate. The graph structure is defined by:

  • NextStepId on each step (linear sequencing)
  • ConditionRule.BranchEntryStepId on condition steps (conditional branching)
  • BranchEntryStepIds on parallel steps (parallel forking)
  • LoopBodyDefinition.Steps on loop steps (nested sub-workflow)

The WorkflowDefinition constructor receives all steps and validates the complete graph in one pass. The aggregate is never in an invalid state.

8.5 Template Reference Validation Rules

Each step's input mappings can contain template references like {{stepA.fieldName}}. These must be validated at design time against the following rules.

8.5.1 Guaranteed Completion Rule

A step can only reference outputs from steps that are guaranteed to have completed before it executes:

Reference Rules by Position

Top-level linear step: Can reference any step earlier in the top-level sequence.

Step inside a condition branch: Can reference steps upstream of the ConditionStepDefinition in the parent sequence, and earlier steps within the same branch. CANNOT reference steps in other condition branches (they don't execute).

Step inside a parallel branch: Can reference steps upstream of the ParallelStepDefinition in the parent sequence, and earlier steps within the same branch. CANNOT reference steps in sibling parallel branches (they may not have completed, or may fail).

The merge step (step after a ParallelStepDefinition): Can reference all steps upstream of the parallel step AND steps from all parallel branches (all branches are guaranteed complete before the merge step runs).

Step inside a loop body: Can reference steps upstream of the LoopStepDefinition in the parent workflow, the loop iteration item (via {{trigger.fieldName}}), and earlier steps within the loop body. CANNOT reference steps downstream of the loop in the parent workflow.

8.5.2 Field Existence Rule

A template reference {{stepA.fieldName}} must point to a field that exists in stepA's declared OutputSchema. If stepA has output schema {messageId: string, timestamp: string}, then {{stepA.timestamp}} is valid but {{stepA.channel}} is rejected at design time.

8.5.3 Null Safety for Parallel Branches

If a step after a parallel merge references {{branchStep.field}} where branchStep is inside a parallel branch, the reference is valid (the branch is guaranteed to have completed). However, if branchStep was skipped due to a failure with Skip strategy, the value will be null at runtime. The design-time validator should warn (not error) about this possibility. The runtime template resolver treats null references as empty strings or null values depending on the target field type.

8.6 DAG Validation Rules

All performed in the WorkflowDefinition constructor:

  1. Exactly one trigger: The workflow must have exactly one TriggerStepDefinition and it must be the entry point.
  2. No cycles: The graph formed by NextStepId links and branch entry points must be acyclic.
  3. All references exist: Every StepId referenced in NextStepId, ConditionRule.BranchEntryStepId, FallbackBranchEntryStepId, and BranchEntryStepIds must correspond to a step in the workflow.
  4. All steps reachable: Every step must be reachable from the trigger by following the graph.
  5. Template references valid: Every template reference must satisfy the Guaranteed Completion Rule (8.5.1) and the Field Existence Rule (8.5.2).
  6. Branch last steps: The last step in every condition branch and parallel branch must have NextStepId = null.
  7. Condition rules: A ConditionStepDefinition must have at least one ConditionRule.
  8. Parallel branches: A ParallelStepDefinition must have at least two BranchEntryStepIds.
  9. Loop source type: The SourceArray template reference on a LoopStepDefinition must point to a field whose type is array in the upstream step's output schema.
  10. Output schemas required: TriggerStepDefinition and ActionStepDefinition must have non-null OutputSchema.

8.7 Execution Semantics

8.7.1 Condition Execution

  1. Evaluate each rule's condition expression top-to-bottom using outputs from completed upstream steps
  2. The first rule that evaluates to true wins
  3. If no rule matches and a fallback exists, take the fallback branch
  4. If no rule matches and no fallback exists, fail the workflow
  5. Execute the winning branch's step chain
  6. When the branch's last step completes (its NextStepId is null), continue to the condition step's NextStepId

8.7.2 Parallel Execution

  1. Create a StepExecution for the first step of every branch simultaneously
  2. Each branch executes independently
  3. When a branch's last step completes, check if all branches are done
  4. If all branches are done, continue to the parallel step's NextStepId (merge point)
  5. If a branch step fails with Stop strategy: cancel all other running branches, fail the workflow
  6. If a branch step fails with Skip strategy: the step is skipped with null output, the branch continues

8.7.3 Loop Execution

  1. Resolve the SourceArray template reference to get the array
  2. Collect upstream step outputs as ParentContext
  3. For each item in the array, create a child WorkflowExecution with the body's definition snapshot, the iteration item as trigger output, and the ParentContext
  4. Execute iterations according to ConcurrencyMode:
    • Sequential: one at a time, in order
    • Parallel(maxN): up to N concurrent iterations, respecting rate limits
  5. If an iteration fails:
    • StopAll: cancel all remaining/running iterations, fail the loop step
    • Skip: record null for that iteration's result, continue
  6. When all iterations complete, the loop step's output is an array of all iteration results
  7. Continue to the loop step's NextStepId

9. Webhook Triggers & Deduplication

External services can trigger workflows via webhook. The platform exposes a unique URL per workflow (e.g., https://platform.com/webhooks/{workflowId}). The external service sends HTTP POST requests to this URL.

Webhooks are unreliable — the sender might retry, delivering the same event multiple times. The system must deduplicate webhook deliveries within a 5-minute window using an idempotency key. Additionally, webhook payloads must be validated against a user-defined JSON schema before triggering the workflow. Invalid payloads are logged but do not create executions.

10. Execution Timeout & Cost Budgets

Each workflow has a maximum execution timeout (default 30 minutes, configurable up to 24 hours for long-running data pipelines).

Each workspace has a monthly execution budget measured in total step executions:

  • Free workspace: 1,000 step executions/month
  • Paid workspace: 50,000 step executions/month

When the budget is exhausted, scheduled triggers stop firing. Webhook triggers still create executions that immediately fail with a "budget exceeded" error (so the webhook sender gets feedback). Budget tracking must be accurate under concurrent execution — 20 workflows finishing simultaneously must not overshoot the budget.

11. Secrets & Variable Scoping

Workspaces have environment variables (e.g., API_KEY, DATABASE_URL) that can be referenced in step configurations using {{env.VARIABLE_NAME}}. Variables have scopes — narrower scope wins:

  1. Workspace-level — shared by all workflows
  2. Workflow-level — overrides workspace defaults for one workflow
  3. Execution-level — passed in at trigger time (e.g., webhook payload fields)

Secrets are environment variables marked as sensitive. They must never appear in execution logs — the system must redact them from step outputs before persisting.

12. Integration Definition Framework

Integrations are not hardcoded. The platform supports a plugin architecture where each integration is defined declaratively. An integration definition contains:

12.1 Authentication Spec

The type of auth (OAuth2, API key, Basic Auth, database connection, custom token refresh), the endpoints for auth flow, token refresh logic, and what headers/query params the token gets attached to. OAuth2 has variations: authorization code flow, client credentials flow, PKCE. Some services require per-store OAuth (Shopify). Others use workspace-level tokens (Slack). Database integrations define connection fields per DBMS (PostgreSQL: host/port/database/username/password/sslMode; MongoDB: connectionUri; SQL Server: server/database/authMethod).

12.2 Action Catalog

Each integration exposes a set of actions. Each action has an input schema, an output schema, and a protocol spec. The protocol spec includes a protocol field that routes to the correct executor:

  • http — method, URL template, headers, body template (covers REST, GraphQL, SOAP)
  • database — dbType, query template
  • smtp — to, subject, body templates

The execution engine reads the protocol spec and dispatches to the correct executor. Adding a new integration means writing a JSON definition, not code.

12.3 Trigger Catalog

Each integration can expose triggers. Two types:

  • Webhook-based: external service pushes events to the platform's endpoint
  • Polling-based: platform periodically calls the API and diffs against a cursor (timestamp, page token, or row ID). Cursor persists per-workflow, not per-integration.

12.4 Response Mapping

A declarative JSONPath-based mapping that extracts data from the raw API response, detects errors, and normalizes the output to match the declared output schema. One generic mapper handles all integrations.

12.5 Pagination

The integration definition specifies the pagination strategy: cursor-based, offset-based, or page-number-based. The execution engine auto-paginates and aggregates all pages into a single output array.

13. Custom Integrations

Users can create their own integrations within their workspace. Custom integrations are HTTP-only. The user defines:

  • Base URL and auth method (API key, Bearer token, Basic Auth)
  • Actions: HTTP method, endpoint path, request body template, and output schema

Output schema can be defined manually or discovered via a "Test this step" feature that makes a real API call, receives the response, and auto-infers the schema by walking the JSON structure.

Custom integrations follow the exact same execution path as built-in integrations. The execution engine does not distinguish between them.

14. Webhook Management

For webhook-based triggers, the platform manages webhook registration with external services. When a workflow is activated, the platform calls the external service's API to register a webhook URL. When deactivated or deleted, the platform deregisters. If registration fails, the workflow cannot be activated.

Services require webhook signature verification — each using different schemes (Slack: HMAC-SHA256 with X-Slack-Signature, Stripe: HMAC-SHA256 with Stripe-Signature, etc.). The integration definition declares the verification method.

Webhook URLs are unique per workflow. Two workflows triggering on the same event get separate webhook registrations and separate incoming URLs.

15. OAuth Token Lifecycle

OAuth tokens expire. The platform must proactively refresh tokens before they expire. If a refresh fails, all workflows using that integration must pause, and the user must be notified to re-authorize.

Critical: Token refresh must be serialized per integration instance. If 10 workflows simultaneously discover the token is expired, they must not all attempt to refresh concurrently. Some providers rotate the refresh token on each use — concurrent refresh attempts cause a race condition where the first refresh invalidates the token the second is using.

16. Rate Limit Discovery & Adaptation

Different APIs express rate limits differently. The integration definition declares the rate limit strategy:

  • Header-based: read remaining/reset headers from every response and adjust (Slack, GitHub)
  • Reactive: handle 429s with exponential backoff (Notion)
  • Configured: hardcoded limit set by the user

The rate limiter adapts in real-time — if the response says "3 requests remaining, resets in 45 seconds," queue the next requests accordingly. Some APIs have multiple rate limit tiers (Shopify: per-minute + daily limits).

Architecture & Performance Pressure

Pressure 1 — Execution Engine Hot Path

A large workspace has 200 active workflows, 50 scheduled to run every minute. That's 400 step executions per minute sustained, with spikes of 50 concurrent workflow starts. The execution engine must scale horizontally. WorkflowExecution and ActionExecution can be processed by different workers.

Pressure 2 — Rate Limiter Under Contention

30 workflows in the same workspace all use Slack (50 requests/minute limit). During a spike, 30 workflows each try to send a Slack message within 10 seconds. The rate limiter must queue excess requests and drain at the allowed rate. If queued requests exceed the step's timeout, the step fails.

Pressure 3 — Execution Log Query Performance

100,000 executions per month per workspace. The UI must return paginated, filtered results within 500ms. Execution logs may belong in a separate read-model database.

Pressure 4 — Workflow Version Migration

A workflow edited 47 times should garbage-collect versions that have no running executions and are not the current active version.

Pressure 5 — For-Each with 50,000 Items

A loop iterating 50,000 times with max concurrency 10 and external API rate limit 100/minute runs for ~8 hours. The execution must be resumable — if the worker crashes at iteration 23,000, it picks up without re-processing. Requires checkpointing.

Pressure 6 — Integration Hot-Swap

Updated integration definitions must not break running executions. Running executions use the old definition. ActionExecution carries a snapshot of the integration definition.

Pressure 7 — Multi-Tenant Token Isolation

10,000 workspaces with encrypted-at-rest tokens. Token refresh for Workspace A must never leak to or corrupt Workspace B's token.

Pressure 8 — Observability per Integration

Cross-tenant read model showing per-integration-type error rate, latency, rate limit headroom, and token health. A Slack outage should be detectable within 2 minutes. Must not expose tenant-specific data.
