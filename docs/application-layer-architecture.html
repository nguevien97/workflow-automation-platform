1. Domain Layer Prerequisites

Before building the application layer, these domain changes are required:

1.1 Missing Domain Events Critical

WorkflowExecution.Cancel(), Suspend(), Resume() change state but emit no events. The application layer needs these to cascade state changes across aggregates and update read models.

Fix: Add WorkflowCancelledEvent, WorkflowSuspendedEvent, WorkflowResumedEvent.

1.2 DateTime.UtcNow Coupling Critical

38 occurrences of DateTime.UtcNow across aggregates, entities, and events. Makes deterministic testing impossible.

Fix: Use .NET's built-in TimeProvider abstraction. Pass it to aggregate methods. Event convenience constructors already accept DateTime — wire them to the provider.

1.3 RejectionRecord Has No Identity Persistence Blocker

Mutable (has MarkSupersededBy()), stored in List<>, but no ID. EF Core cannot track it.

Fix: Add RejectionRecordId, make it an Entity like LoopIteration.

1.4 Aggregate Versioning for Optimistic Concurrency

When two processes try to modify the same WorkflowExecution concurrently (e.g., two parallel branches completing), one must fail and retry.

Fix: Add Version property to AggregateRoot<TId>, incremented on each save.

1.5 IRepository Extension

Current interface only has GetByIdAsync, AddAsync, UpdateAsync. Need per-aggregate repository interfaces with query methods, and IUnitOfWork for transaction coordination.

2. Application Layer Responsibilities

The application layer sits between the API/presentation layer and the domain:

ResponsibilityWhat It Means
Use case orchestrationReceive a command, load aggregates, invoke domain methods, persist changes
Domain event dispatchAfter saving, dispatch events to handlers that coordinate cross-aggregate flows
Event transformationMap domain events to integration events for cross-process communication
Transaction managementDefine what gets saved atomically
AuthorizationVerify the caller has permission to perform the operation
ValidationValidate command structure before touching the domain
Read model maintenanceProject domain events into query-optimized data structures
Integration orchestrationCoordinate calls to external systems triggered by domain events
What it does NOT do
  • Business rules (those stay in domain aggregates)
  • Infrastructure details (those live in the infrastructure layer)
  • HTTP/gRPC specifics (those live in the API layer)

4. Command & Query Design (CQRS)

Commands (Write Side)

Commands represent user intent. Each command has exactly one handler.

WorkflowDefinition Commands

CommandWhat It Does
CreateWorkflowDefinitionCommandValidate + persist a new workflow definition
PublishWorkflowVersionCommandMark a definition as Published (available for execution)
DeprecateWorkflowVersionCommandPrevent new executions against this version

WorkflowExecution Commands

CommandWhat It Does
StartWorkflowCommandMap definition to snapshot, create WorkflowExecution, call Start()
ApproveReviewStepCommandLoad execution, call ApproveReviewStep(), advance graph
RejectReviewStepCommandLoad execution, call RejectReviewStep(), rewind graph
RecordIntegrationResultCommandLoad ActionExecution, record success/failure, trigger completion flow
CancelWorkflowCommandLoad execution, call Cancel(), cascade to children
SuspendWorkflowCommandPause execution (e.g., rate limit hit)
ResumeWorkflowCommandResume suspended execution

Command Handler Structure

1. Validate command (FluentValidation)
2. Authorize caller
3. Load aggregate(s) from repository
4. Call domain method(s)
5. Save via UnitOfWork (triggers event dispatch + outbox write)
6. Return result

Queries (Read Side)

Queries never modify state. They read from the domain model (Stage 1) or from projected read models (Stage 2+).

QueryReturns
GetWorkflowExecutionQueryExecution status, step statuses, current position in graph
GetRunningExecutionsQueryAll active executions for a workspace (paginated)
GetExecutionHistoryQueryCompleted/failed executions with filtering
GetStepExecutionDetailsQueryInput, output, error, attempts for a specific step
GetPendingReviewsQueryAll executions waiting for review approval

5. Domain Event Dispatching

The Problem

Aggregates collect events via AddDomainEvent(). Nothing dispatches them yet. The dispatch mechanism must handle when events fire relative to persistence, cascading events, and integration event extraction.

Dispatch Algorithm

UnitOfWork.SaveChangesAsync():
  1. Persist all tracked aggregates to DB (do NOT commit yet)
  2. Collect domain events from all tracked aggregates
  3. Clear events from aggregates
  4. For each domain event:
     a. Invoke in-process handlers (MediatR INotificationHandler<T>)
     b. Handlers may load more aggregates, call domain methods, produce new events
     c. Persist any newly modified aggregates
  5. Repeat steps 2-4 until no new events (cap at 10 iterations)
  6. For each domain event, run event mappers:
     a. If mapper returns an integration event, serialize to OutboxMessages table
  7. COMMIT transaction (aggregates + outbox in one atomic operation)
Why Before Commit (Not After)

If a handler for ActionExecutionRequestedEvent creates an ActionExecution aggregate, that aggregate MUST be saved in the same transaction as the WorkflowExecution that emitted the event. If the transaction commits WorkflowExecution with status=Running but the ActionExecution creation fails separately, the workflow is stuck forever.

Cascading Event Example

StartWorkflowCommand:
  handler calls WorkflowExecution.Start()
    emits WorkflowStartedEvent
    internally calls ExecuteStep() which emits ActionExecutionRequestedEvent

Dispatch round 1:
  - WorkflowStartedEvent -> (no handler needed, or read model projection)
  - ActionExecutionRequestedEvent -> handler creates ActionExecution, calls Execute()
    ActionExecution emits IntegrationRequestedEvent

Dispatch round 2:
  - IntegrationRequestedEvent -> mapper produces ExecuteIntegrationCommand for outbox
    (no in-process handler -- this crosses a process boundary)

No more events. Commit.
Result: WorkflowExecution + ActionExecution saved atomically.
        ExecuteIntegrationCommand queued in outbox for async delivery.

6. Domain Events vs Integration Events

This is a critical architectural distinction. Confusing these two concepts leads to either brittle coupling or lost events.

AspectDomain EventIntegration Event
ScopeIn-process, same transactionCross-process, async via message bus
PurposeCoordinate aggregates within bounded contextNotify other processes/services
Defined inDomain projectContracts project (shared)
ContainsStrongly-typed IDs (StepId)Primitive types (Guid, string)
DeliveryIn-memory, synchronousMessage bus, at-least-once
FailureRolls back entire transactionRetried independently, dead-lettered
OrderingGuaranteed (sequential dispatch)Guaranteed per partition key only
StabilityCan change freely on refactorMust be versioned carefully

Naming Convention Encodes Intent

  • *Projection — read model update, no business consequence if delayed

Integration Event Envelope

public record IntegrationEventEnvelope(
    Guid EventId,              // Unique, for idempotency
    string EventType,          // "ExecuteIntegrationCommand" (for routing)
    string PartitionKey,       // WorkflowExecutionId (for ordering)
    string Payload,            // Serialized JSON
    DateTime CreatedAt,
    int Version                // Schema version (for evolution)
);

Version Evolution

When you need to change ExecuteIntegrationCommand (e.g., adding a Priority field): add it with a default value. Old consumers ignore unknown fields. Or introduce V2 — the mapper decides which version to produce. The domain event is unaffected. Domain refactors never break wire contracts.

8. Cross-Aggregate Coordination (Sagas)

The Coordination Chains

Chain 1: Action Step Lifecycle

WorkflowExecution.ExecuteStep(Action)
  -> ActionExecutionRequestedEvent
  -> [Handler] Create ActionExecution, call Execute()
     -> IntegrationRequestedEvent
     -> [Handler/Worker] Call external API
        -> RecordIntegrationResultCommand
        -> [Handler] ActionExecution.RecordIntegrationSucceeded()
           -> ActionCompletedEvent
           -> [Handler] WorkflowExecution.RecordActionCompleted()
              -> StepCompletedEvent
              -> AdvanceOrComplete() -> may emit next ActionExecutionRequestedEvent

Chain 2: Loop Step Lifecycle

WorkflowExecution.ExecuteStep(Loop)
  -> LoopExecutionStartedEvent
  -> [Handler] Create LoopExecution, call Start()
     -> LoopIterationStartedEvent (per iteration, up to MaxConcurrency)
     -> [Handler] Create child WorkflowExecution per iteration
        -> Child runs full Chain 1/2 recursively
        -> Child WorkflowCompletedEvent
        -> [Handler] LoopExecution.RecordIterationCompleted()
           -> When all done: LoopCompletedEvent
              -> [Handler] WorkflowExecution.RecordLoopCompleted()
                 -> AdvanceOrComplete()

Chain 3: Parallel Branches (In-Process)

WorkflowExecution.ExecuteStep(Parallel)
  -> ParallelBranchesForkedEvent
  -> ExecuteStep() for each branch entry (same aggregate)
  -> Each branch runs Chain 1/2 independently
  -> Each branch completion checks AreAllParallelBranchesCompleted()
  -> When last branch: ParallelBranchesMergedEvent -> AdvanceOrComplete()

Chain 4: Review Step (Human Pause)

WorkflowExecution.ExecuteStep(Review)
  -> ReviewStepReachedEvent
  -> [Notification Worker] Send approval request (email, Slack, UI)
  -> ... time passes ...
  -> ApproveReviewStepCommand -> AdvanceOrComplete()
  OR
  -> RejectReviewStepCommand -> invalidate range, re-execute from target

Process Manager State

For timeout detection and recovery in Stages 2-3:

ProcessManagerState:
  WorkflowExecutionId
  CurrentStepId
  ExpectedEvent (e.g., "IntegrationResultEvent for ActionExecutionId X")
  TimeoutAt (DeadlineUtc from ActionExecution)
  State (WaitingForIntegration, WaitingForRetry, WaitingForReview, Completed)

9. Cross-Cutting Concerns

Applied uniformly across all command/query handlers via MediatR IPipelineBehavior<TRequest, TResponse>.

9.1 Authorization & Workspace Scoping

Every command carries WorkspaceId and UserId. The authorization behavior verifies the user belongs to the workspace and has the required permission.

PermissionCommands
Workflow.CreateCreateWorkflowDefinition, PublishVersion
Workflow.ExecuteStartWorkflow
Workflow.ManageCancelWorkflow, SuspendWorkflow, ResumeWorkflow
Review.ApproveApproveReviewStep, RejectReviewStep
Execution.ReadAll queries

Multi-tenancy: Every DB query must include WHERE WorkspaceId = @workspaceId.

9.2 Validation Pipeline

Application validation (FluentValidation, before domain): command structure, reference checks, state preconditions.

Domain validation (inside aggregates): guard methods, state machine checks.

Error Mapping

InvalidOperationException (domain guard)
  -> CommandResult.Failure(DomainErrorCode, message)
    -> API: HTTP 409 Conflict or 422 Unprocessable Entity

ValidationException (FluentValidation)
  -> CommandResult.ValidationFailure(errors[])
    -> API: HTTP 400 Bad Request

UnauthorizedException
  -> API: HTTP 403 Forbidden

9.3 Logging & Correlation

Workflow executions span multiple commands over hours/days. Every log entry needs:

  • WorkflowExecutionId — the primary correlation ID
  • StepId — which step is being processed
  • ActionExecutionId / LoopExecutionId — if applicable
  • CommandType — which command triggered this log

For distributed mode: the CorrelationId must propagate through message bus headers.

9.4 Transaction Management

TransactionBehavior<T>:
  Begin transaction
    -> Run handler (load aggregates, call domain methods)
    -> UnitOfWork.SaveChangesAsync() (persist + dispatch events + write outbox)
    -> If success: commit
    -> If exception: rollback (all aggregate changes + outbox undone)

Query handlers do NOT get a transaction behavior (read-only).

9.5 Idempotency

Command-level: Each command carries an optional IdempotencyKey. The behavior checks against IIdempotencyStore and returns cached result if already processed.

Event-handler-level: Domain aggregate guards prevent double-processing. Handler catches guard exception and treats it as "already processed."

Integration callback: RecordIntegrationResultCommand includes ActionExecutionId + AttemptNumber. ActionExecution validates the attempt number matches.

9.6 Rate Limiting & Throttling

  • Per-workspace rate limiting: prevent one workspace from starving others
  • Integration rate limiting: external APIs have rate limits, enforce before dispatching
  • Global back-pressure: reject new StartWorkflow commands with 429 when overloaded

The MediatR Pipeline Stack

Request arrives
  -> LoggingBehavior (set correlation ID, log entry/exit)
  -> ValidationBehavior (FluentValidation on command fields)
  -> AuthorizationBehavior (check user permissions + workspace scope)
  -> IdempotencyBehavior (short-circuit if already processed)
  -> TransactionBehavior (begin/commit/rollback)
     -> Handler executes
        -> Domain events dispatched (within transaction)
        -> Integration events written to outbox (within transaction)
  -> Response returned

11. Read Models & Projections

Why Separate Read Models

Domain aggregates are optimized for writes. Loading a full WorkflowExecution with its snapshot, step executions, and rejection history just to show a status badge is wasteful.

Read Model Tables

WorkflowExecutionView:
  Id, WorkspaceId, WorkflowName, Status, StartedAt, CompletedAt,
  CurrentStepName, TotalSteps, CompletedSteps, FailedSteps, StartedBy

StepExecutionView:
  Id, WorkflowExecutionId, StepName, StepType, Status,
  StartedAt, CompletedAt, Error, AttemptCount

PendingReviewView:
  WorkflowExecutionId, StepId, StepName, ReachedAt, WorkflowName

ActiveLoopView:
  LoopExecutionId, WorkflowExecutionId, StepName,
  TotalIterations, CompletedIterations, FailedIterations

Projection Strategy

Stage 1: Query domain model directly via EF Core. Good enough for <100 concurrent workflows.

Stage 2+: Project from domain events into denormalized read model tables. Eventually consistent in distributed mode (1-2s lag acceptable for dashboards).

13. Stage 1: Single-Process Monolith

One ASP.NET process. API endpoints + domain event handlers + background jobs all in-process.

StartWorkflowCommand
  -> Handler creates WorkflowExecution, calls Start()
  -> SaveChangesAsync dispatches events:
     -> ActionExecutionRequestedHandler creates ActionExecution, calls Execute()
     -> IntegrationRequestedEvent handled by in-process integration caller
     -> RecordIntegrationSucceeded -> ActionCompletedEvent
     -> ActionCompletedHandler advances WorkflowExecution
     -> Next step's events dispatched...
  -> All saved in one transaction
  -> Response returned to caller
Integration Calls Block

External API calls can take seconds. Options: (1) Accept it — for <100 workflows, a few seconds is fine. (2) Use IHostedService with an in-memory queue — adds ~50 lines of code.

What You Build

  • All command/query handlers
  • All domain event handlers
  • MediatR pipeline behaviors (validation, logging, transaction)
  • EF Core persistence with repositories
  • Direct queries for read endpoints
  • In-memory IDomainEventDispatcher

What You Skip

  • Outbox table, message bus
  • Integration event contracts & event mappers
  • Read model projections
  • Idempotency store (guards are sufficient)

14. Stage 2: Outbox + Background Workers

When You Need This

  • Integration calls are too slow to block requests
  • You need reliability (survive process restarts)
  • You want to separate notification delivery from core execution

Changes from Stage 1

Add Outbox

OutboxMessages table in the same DB. SaveChangesAsync writes integration events to outbox atomically. Background OutboxDispatcher polls and publishes.

Separate Integration Calls

IntegrationRequestedEvent → mapper writes ExecuteIntegrationCommand to outbox. IntegrationWorker (IHostedService) processes it, calls external API, dispatches result back.

Separate Notifications

ReviewStepReachedEvent → mapper writes ReviewPendingNotification to outbox. NotificationWorker processes it.

Coordination Events Stay In-Process

ActionCompletedEvent, LoopCompletedEvent, etc. still dispatched via MediatR within SaveChangesAsync. No message bus needed for these yet.

What You Add

  • OutboxMessages table + OutboxDispatcher
  • Event mappers for integration + notification events
  • IntegrationWorker background service
  • NotificationWorker background service
  • Idempotency checks on RecordIntegrationResultCommand

Key Changes from Stage 2

ALL events go through outbox + message bus. Coordination events become integration events in the Contracts project. The Orchestration Worker consumes them.

Partitioning: All events partitioned by WorkflowExecutionId. Guarantees ordering per workflow. Different workflows processed in parallel. LoopExecution events partitioned by parent WorkflowExecutionId.

Idempotency everywhere: Every handler checks aggregate state. IIdempotencyStore for command deduplication. Message ACK only after successful save.

Saga persistence: Process manager state persisted to DB. On restart, unfinished sagas resume. Timeout Monitor checks for stale sagas.

Scaling the Orchestration Worker

Horizontal scaling via partition assignment: Message bus has N partitions (e.g., 64). Each worker instance handles a subset. 4 instances → each handles ~16 partitions → ~250 concurrent workflows each. Adding instances rebalances automatically.

Why partition by WorkflowExecutionId: All events for one workflow go to the same partition → same worker instance. No locking needed. Parallel branches completing "simultaneously" arrive sequentially within the partition.

Scaling the Integration Worker

Independent of orchestration. Subscribes only to ExecuteIntegrationCommand. Can have 10x more instances. Stateless: receives command, calls API, publishes result.

Per-integration rate limiting: Token bucket per IntegrationId, shared across instances via Redis.

16. Concurrency & Scaling Considerations

Concurrency Scenarios at Scale

Scenario 1: Two parallel branches complete simultaneously

Both trigger ActionCompletedMessage for the same WorkflowExecution. Partitioned by WorkflowExecutionId → same partition → processed sequentially. First message advances, second triggers merge.

Scenario 2: Loop with 100 iterations, MaxConcurrency=10

10 child workflows start immediately, each on its own partition. Children run independently at full parallelism. Completions flow back to LoopExecution (parent's partition), start next pending iterations.

Scenario 3: 1000 concurrent workflows, 10 steps each

1000 WorkflowExecutionIds across 64 partitions. ~16 per partition. 4 Orchestration Workers, each handles ~250 workflows. Integration Worker handles ~10,000 API calls (8-10 instances).

Scenario 4: Nested loops (loop inside loop)

Outer: 10 iterations. Inner: 50 each. Total child workflows: 500. Each child on its own partition. Inner loop completions flow back to parent LoopExecution. Deep nesting works because each level has its own aggregate.

17. Failure Modes & Recovery

FailureImpactRecovery
Message bus down Outbox writes succeed. Execution pauses. OutboxDispatcher retries on polling interval. No data loss. API continues accepting commands.
Orchestration Worker crashes Message not acknowledged. Bus redelivers to another instance. Idempotent handlers ensure correctness.
Integration Worker crashes after API call External call may have succeeded. ExecuteIntegrationCommand redelivered, worker calls API again. External API must be idempotent or use correlation ID.
Database failover In-flight transactions roll back. Connection retry with exponential backoff. Outbox dispatcher pauses, resumes. Projector catches up.
Stuck workflows No event arrives past deadline. Timeout Monitor detects via ProcessManagerState WHERE TimeoutAt < now. Fires RecordDeadlineExceeded. Workflow fails or skips per FailureStrategy.
Poison messages Handler fails repeatedly. Moved to dead-letter queue after 3 attempts. Alert to operations. Manual replay after fix.
